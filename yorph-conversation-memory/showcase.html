Conversation Memory
Cursor / Claude Code Skill · by Yorph AI

After compaction, your agent forgets why it made every decision

Long coding sessions fill the context window. When it compacts, the agent keeps a vague summary but loses specifics — which alternatives were rejected and why, what edge cases were discovered, what the user actually said they wanted. It re-proposes dead ideas, re-derives known answers, and contradicts earlier decisions. Solution: a manually triggered checkpoint that writes the conversation's reasoning and decisions to disk as structured topic files, indexed by keywords in the skill description so the agent can find them post-compaction without guessing.

Example 01
Decision Recall
50 turns in, the user asks about a rejected approach

Early in conversation (turn 12)

User
Could we use Redis for the caching layer?
Agent
Redis adds ~800ms cold-start latency in Lambda. Since every lookup happens within a single invocation, an in-process LRU cache gives sub-millisecond reads with zero infrastructure. I'd go with that unless you need cross-invocation persistence.
User
Makes sense, let's do the LRU cache.
⚡ Context window compacts — 40 turns compressed into summary

Later in conversation (turn 58)

User
Actually, could we revisit Redis? My team lead is asking about it.
❌ Without Memory
compacted summary only
"We discussed caching earlier. I believe we went with a local cache, though I don't recall the specific trade-offs."
The agent lost the rejection rationale. It can't tell the user why Redis was ruled out — forcing a redundant re-evaluation, or worse, re-proposing Redis without flagging the Lambda cold-start problem.
✓ With Conversation Memory
reads .memory/api-build/decisions.md
"We rejected Redis because of ~800ms cold-start latency in Lambda. The in-process LRU cache avoids that since all lookups are within a single invocation."
The agent gives the exact rationale — the user's team lead gets a clear, defensible answer without re-doing the analysis.

How the Agent Finds the Right File
Keywords in the skill description act as a search index

The skill description — visible in the system prompt every turn — contains session names and aggregated keywords. When the user mentions a term, the agent matches it to a session. Reading the skill body reveals the per-topic keyword table, pointing to the exact file.

Technique
Self-indexing memory via a dynamic skill description
Three-phase checkpoint: categorize, write, index
Phase 1 · Categorize

Extract ephemeral knowledge

The agent reviews the conversation and identifies what only exists in the chat — decisions, rejected alternatives, user preferences, its own reasoning. Anything already written to files (code, configs, docs) is skipped.

Phase 2 · Write

Structured topic files on disk

Each topic becomes a markdown file in .memory/{session}/. Files are focused: one topic per file, with timestamps and specific values. The agent reads these with a single tool call when it needs a detail.

Phase 3 · Index

Keywords into the skill description

The SKILL.md description is rewritten with session names and keywords. This description appears in the system prompt every turn — including after compaction — so the agent always knows what memory is available and can match user queries to topics.

Caveats
Caveats

Manual trigger only

There's no way to detect when compaction is about to happen. The user must say "checkpoint" — if they forget, the knowledge is lost. Compaction has no pre-signal exposed to the agent.

Self-modifying skill file

The SKILL.md rewrites its own description at every checkpoint. If the rewrite has a bug, the skill's description breaks. Static instructions live in a separate file to limit blast radius.

Keywords may miss

The agent picks keywords at checkpoint time. If the user later asks about a topic using different terminology, the keyword match might fail. Grep on the memory directory is the fallback.

No cross-conversation isolation

There's no host-provided conversation ID. Sessions are user-named, which works for sequential conversations but can collide if two people checkpoint to the same session name simultaneously.
