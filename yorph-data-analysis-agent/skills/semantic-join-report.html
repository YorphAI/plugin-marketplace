Most analytics pipelines quietly fail when data is messy. A query for "chocolate" returns nothing for rows labelled "drk choc bar" or "mlk choc". Two tables that clearly share records can't be joined because no clean key exists. Customer complaints reference products that don't match any column value.

The fix isn't cleaning the data by hand. It's asking an LLM to extract structure from the text — once, cheaply, up front — and then doing the rest with ordinary code.

The approach demonstrated here is inspired by Featurized-Decomposition Join (Zeighami, Shankar, Parameswaran — UC Berkeley, 2025), which showed up to 10× cost reduction in LLM-powered data joins by extracting features first rather than running pair-wise comparisons.

The same principle applies to enrichment, aggregation, and cross-table linking — any task where the answer lives inside messy text.

The Method
Three phases, linear cost.
The expensive part — calling an LLM — happens once per row, not once per pair. Everything after that is plain code.
Phase 01
Inspect & Propose

Sample the dataset. Ask an LLM what structured columns are latent in the text fields given the user's goal. It proposes the extraction schema — you don't hardcode anything.

1 LLM call on a sample
Phase 02
Extract at Scale

Run the extraction over every row in a single batched pass. Canonical names, categories, parsed values — all become new columns. Prefer regex and normalization where possible; use LLM only where meaning is genuinely ambiguous.

O(n) · one pass per row
Phase 03
Answer with Code

With clean structured columns, the user's actual question — aggregate, join, filter, deduplicate — becomes a standard operation. For cross-table joins, use extracted features to prune candidate pairs before any LLM confirmation step.

The Takeaway
Extraction first. Analysis second.
The pattern

Ask an LLM what structure is latent in your text fields. Extract it once per row. Then use ordinary code for everything else — groupby, join, filter, aggregate.

The cost model

O(n) LLM calls, not O(n²). For a 1,000-row table joined to a 500-row catalog, that's 1,500 calls instead of 500,000. The filter step reduces LLM confirmations by 70%+ on top of that.

What it unlocks

Joins with no shared key. Aggregations over categories that don't exist yet. Linking free-text to structured tables. Analysis that was previously only possible after expensive manual data cleaning.
