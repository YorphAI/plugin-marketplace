The Technique
Extract once. Answer with code.
The LLM is used twice: once on a sample to propose an extraction schema, and once to generate concept mappings that get applied to the full dataset in bulk. The rest is standard Python or SQL.
Phase 01
Sample & Propose

The LLM inspects a small sample of your dataset and proposes what structured columns are latent in the text — given your actual goal. No hardcoded schema. The structure comes from the data.

1 LLM call on a sample
Phase 02
Extract at Scale

Run the schema across every row in a single batched pass. Canonical names, categories, parsed tokens — all become new columns. One call per row, not one call per pair.

O(n) · one pass per row
Phase 03
Answer with Code

With clean structured columns, your actual question — aggregate, join, filter, deduplicate — becomes a standard operation. For cross-table joins, extracted features prune candidate pairs before any LLM confirmation step.

Standard Python / SQL
Based on Featurized-Decomposition Join — Zeighami, Shankar, Parameswaran · UC Berkeley · arXiv:2512.05399 · 2025. The paper demonstrated up to 10× cost reduction in LLM-powered joins by extracting decomposition features first rather than running pairwise comparisons.
Why It Scales
The LLM maps concepts.
Code does the rest.

FDJ doesn't call the LLM per row. It uses the LLM to generate concept mappings from a sample — {"mlk choc": "milk chocolate", "drk choc bar": "dark chocolate bar", ...} — then applies those mappings to the full dataset with standard pandas operations. The LLM call count stays roughly constant regardless of dataset size. Compare that to 51,000 pairwise LLM comparisons for the same 850-row × 60-product join done naïvely.
